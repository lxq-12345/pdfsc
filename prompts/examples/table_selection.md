# Table Conversion - Selection Rules

## 表格三层次快速判断指南

### 第一步：识别数据维度

数据维度 = 表格中独立的信息类别数

#### 示例1：符号说明表
```
符号 ← 维度1
含义 ← 维度2
总计 2 维
```
**判断**: 2维 → 第一层次

---

#### 示例2：故障排查表
```
现象      ← 维度1（问题分类）
原因      ← 维度2（问题属性）
解决方案  ← 维度3（操作类别）
总计 3 维
```
**判断**: 3维 → 第二层次 ⭐

---

#### 示例3：型号规格表
```
型号      ← 维度1（实体）
形态      ← 维度2（属性类别1）
处理器    ← 维度3（属性类别2）
内存      ← 维度4（属性类别3）
功耗      ← 维度5（属性类别4）
总计 5 维+
```
**判断**: 4维+ → 第三层次

---

### 第二步：查表确定转换方式

| 维度数 | 层次 | 转换方式 | 使用场景 |
|--------|------|---------|---------|
| 2维 | 第一层次 | 语义列表 | 符号说明、术语解释、简单对应 |
| 3维 | 第二层次⭐ | 完整句式 | 故障排查、配置说明、操作指南 |
| 4维+ | 第三层次 | 列表加定义 | 产品对比、规格参数、详细清单 |

---

### 第三步：特殊情况处理

#### 情况1：多行数据的表格
```
原格式（多行）：
情况1 | 步骤很长，需要换行...
      | 的处理方式

转换：使用第二层次（完整句式）
→ 避免Markdown表格的换行问题
```

#### 情况2：混合维度
```
数据既有2维又有3维的内容
→ 判断主体：查表中一行最多有几个信息类别
→ 按多维度来处理
```

#### 情况3：隐含维度
```
表格虽然看起来只有2列，但有隐含的分类
示例：
  硬件要求
  - CPU: Intel i7
  - 内存: 16GB

这其实是 3维（硬件类别+属性+值）
→ 可用第二或第三层次处理
```

---

### 快速参考流程

```
遇到表格
  ↓
数1：是不是只有两列？→ 是 → 可能是2维或隐含维度
  ↓ 不是
数2：表格有几列？
  ↓ 3列 → 3维 → 使用第二层次 ✅
  ↓ 4列+ → 4维+ → 使用第三层次
  ↓ 2列 → 继续判断
判断隐含维度：列名是分类吗？
  ↓ 是 → 至少3维 → 使用第二或第三层次
  ↓ 否 → 2维 → 使用第一层次
```

## 质量检查清单

转换完成后检查：

- [ ] 没有任何传统Markdown表格残留（`|---|`格式）
- [ ] 所有数据都被转换为自然语言形式
- [ ] 2维数据用了语义列表
- [ ] 3维数据用了完整句式（推荐）
- [ ] 4维+数据用了列表加定义
- [ ] 转换后的内容清晰易读
- [ ] 没有数据丢失或被隐藏
- [ ] RAG理解无歧义（尤其是关键参数）
