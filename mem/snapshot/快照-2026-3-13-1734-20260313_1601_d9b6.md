# 会话补充快照

## 记录目的
用户要求将本轮关于“Claude终端项目连续记忆机制”的讨论结论明确记录到会话快照中，作为后续理解与执行依据。

## 核心结论
1. 新开的 Claude 终端默认不直接继承前一个终端的对话历史。
2. 项目连续性主要依赖项目文件体系来重建，而不是依赖模型自动记住上一次对话。
3. `CLAUDE.md` 是项目级入口文件，负责规定读取顺序、执行优先级和恢复路径。
4. 常规启动链路通常是：`会话管理协议.md` → `plan/项目规划.md` → `skills/Markdown转换规范.md` → `mem` 下最新备忘文件。
5. `mem/snapshot/` 下的快照文件不是每次启动都固定读取，只有在启动检查发现未归档快照、需要恢复中断上下文时，才进入读取链路。
6. 因此，这套机制最准确的定义是：项目连续性 = 文件驱动的上下文重建机制；其中快照恢复 = 条件触发的补充分支，而不是固定主链路。

## 讨论中的澄清点
1. “唯一记忆来源是文件”这句话在“项目连续性恢复”场景下基本成立，但不适用于描述当前会话的全部上下文来源。
2. 当前工作目录、终端状态、工作区文件结构等属于环境上下文，不是由 `CLAUDE.md` 生成，但会影响当前会话。
3. 在本项目当前状态下，平台级记忆空间未见有效内容，因此项目连续性的主要来源确实是项目根目录内、并由 `CLAUDE.md` 指导读取的文件体系。

## 与网页 Claude 讨论后的修正
1. 网页 Claude 认可“文件驱动的上下文重建机制”这一表述。
2. 网页 Claude 补充指出：快照文件的读取属于条件触发流程，而非每次启动的固定步骤。
3. 双方最终结论一致，分歧点仅在“快照是否属于固定主链路”。
