## 📋 文档结构

### 1. Frontmatter元数据（必须）

所有Markdown文件必须以YAML格式的Frontmatter开头：

```yaml
---
title: 文档完整标题
category: 分类名称
type: 文档类型 (如：指南、手册、参考、概述等)
model: 适用型号 (如：TaiShan 200 型号2280)
version: 版本号 (从原PDF提取)
date: 2026-02-28
keywords: [关键词1, 关键词2, 关键词3]
source_pdf: 原始PDF文件名
---
```

#### Frontmatter字段说明

| 字段 | 说明 | 例子 |
|------|------|------|
| title | 文档完整标题，尽可能保留原PDF标题 | TaiShan 200 服务器 快速安装指南（型号2280） |
| category | 文档分类（对应工作规划中的7类） | 安装配置 |
| type | 文档类型 | 安装指南 |
| model | 适用的服务器型号 | TaiShan 200 型号2280 |
| version | 原始PDF的版本号 | 10 |
| date | 转换日期 | 2026-02-28 |
| keywords | 关键词数组（3-5个关键词） | [安装, 快速开始, 硬件配置] |
| source_pdf | 原始PDF文件的完整名称 | TaiShan 200 服务器 快速安装指南 (型号2280) 10.pdf |

## 📌 标题层级规范

### 标准层级结构

```
# 主标题（一级标题）- 文档标题
## 一级章节（二级标题）- 如"第1章 简介"
### 二级小节（三级标题）- 如"1.1 环境参数"
#### 三级内容点（四级标题）- 如具体小项 (可选，尽量避免使用五级及以上)
```

### 标题规范要求

1. **一级标题（#）**
   - 每个Markdown文件有且只有1个一级标题
   - 位置：在Frontmatter之后，作为文档标题
   - 格式：`# 文档完整标题`
   - 示例：`# TaiShan 200 服务器 快速安装指南（型号2280）`

2. **二级标题（##）**
   - 用于文档的主要章节
   - 格式：`## 章节名 (可选：序号前缀)`
   - 示例：
     - `## 文档信息`
     - `## 1 简介`
     - `## 2 准备工具`

3. **三级标题（###）**
   - 用于小节
   - 格式：`### 小节名`
   - 示例：
     - `### 1.1 环境参数`
     - `### 符号说明`

4. **四级标题（####）**
   - 用于详细小项（非必需）
   - 示例：
     - `#### 温度`
     - `#### 湿度`

### 禁止事项
- ❌ 不使用五级及以上标题（#####及以上）
- ❌ 不使用多个一级标题
- ❌ 标题前后不加额外符号（如 "### 标题 ###"）

## 📝 列表规范

### 无序列表

使用 `-` 或 `*` 作为列表符（保持一致）

```markdown
- 列表项1
- 列表项2
  - 子列表项2.1
  - 子列表项2.2
- 列表项3
```

**规范要求**：
- 列表符后有一个空格
- 子列表使用2个空格缩进
- 子列表的子列表使用4个空格缩进

### 有序列表

使用 `1.`, `2.` 等数字序号

```markdown
1. 第一步
2. 第二步
   1. 小步骤2.1
   2. 小步骤2.2
3. 第三步
```

**规范要求**：
- 数字后跟一个点和一个空格
- 子列表保持相同的缩进规则

### 定义列表

使用 `-` 加冒号的格式：

```markdown
- 术语1：术语1的定义和说明
- 术语2：术语2的定义和说明
  - 子项2.1：用于分类或细化
  - 子项2.2：用于分类或细化
```

## 📊 表格规范（采用语义列表完整句式）

### ⚠️ 核心原则：禁用传统Markdown表格

**弃用原因**：
- 多行内容一旦换行，表格结构就会崩溃
- 列数过多（>5列）时横向溢出，可读性极差
- 源码维护困难，列宽不均时难以对齐
- RAG向量化后语义理解差，模型难以理解表格的实际含义

**替代方案**：采用**分层次的语义列表完整句式**

---

### 转换方法：三层次递进

#### 第一层次 - 语义列表格式（基础）

使用分类和子项的方式组织表格数据：

```markdown
现象：电源灯不亮
可能原因：电源线未连接 → 解决方法：检查电源线连接
可能原因：电源模块故障 → 解决方法：更换电源模块
可能原因：供电不足 → 解决方法：检查机柜供电容量
```

**适用情况**：简单的对应关系，3行以内

---

#### 第二层次 - 结构化完整句式格式（推荐 ⭐⭐⭐⭐⭐）

使用自然语言的完整句子表达，按属性分行组织，提供最佳RAG语义理解效果：

```markdown
现象：电源灯不亮
电源灯不亮的可能原因及解决方法如下：电源线未连接时，需检查电源线连接；
电源模块故障时，需更换电源模块；供电不足时，需检查机柜供电容量。

现象：启动缓慢
启动缓慢的可能原因及解决方法如下：BIOS自检耗时时，等待或检查硬件；
硬盘故障时，需检查硬盘状态。
```

**结构化分行版本**（更优，用于规格参数等）：

```markdown
#### TaiShan 100 2280（2U双路机架服务器）

TaiShan 100 2280是2U双路机架服务器，搭载2个华为鲲鹏916处理器，支持高性能计算应用。

服务器配置：16个DDR4-2400内存插槽，最大内存容量为512GB。

存储：可支持最多16个3.5英寸或27个2.5英寸的SAS/SATA硬盘或固态硬盘。

RAID：支持0/1/5/6/10/50/60多种级别，并提供超级电容掉电保护。

PCIe扩展：最多可配备5个PCIe 3.0 x8插槽。

网络配置：2个GE电口加2个10GE高速网络接口。
```

**优势**：
- ✅ 完整的自然语言结构，RAG向量化理解最优
- ✅ 属性分行，结构清晰，易于扫读
- ✅ 支持多行内容，无换行问题
- ✅ 源码维护简便
- ✅ 人类阅读最流畅，无符号干扰

**适用情况**：大多数数据表格（故障排查、参数说明、规格对比、对应关系等）

---

#### 第三层次 - 列表加定义格式（复杂数据）

当内容复杂或需要多个维度时，使用列表和定义组合：

```markdown
### 型号对比

TaiShan 200 型号2280
- 形态：2U双路服务器
- 处理器：2x华为鲲鹏920处理器
- 内存：最多32个DIMM，支持DDR4内存
- 最大功耗：2800W
- 集成存储：支持6x3.5英寸或12x2.5英寸硬盘

TaiShan 200 型号2180
- 形态：2U单路服务器
- 处理器：1x华为鲲鹏920处理器
- 内存：最多16个DIMM，支持DDR4内存
- 最大功耗：1600W
- 集成存储：支持4x3.5英寸或8x2.5英寸硬盘
```

**优势**：
- ✅ 支持多层次、多维度的数据
- ✅ 结构清晰，易于扫读和理解
- ✅ RAG理解准确，支持向量化检索
- ✅ 易于维护和扩展

**适用情况**：产品对比、规格参数、功能清单等多维度数据

### 表格规范要求

#### 禁止事项
- ❌ **不使用传统Markdown表格**（包括GitHub Flavored Markdown）
- ❌ 不使用单元格内的复杂符号和换行
- ❌ 不使用对齐符号和复杂表格结构

#### 必须遵循
1. **选择合适的层次**：
   - 简单对应关系 → 语义列表格式
   - 大多数数据表 → 完整句式格式（推荐）
   - 复杂多维数据 → 列表加定义格式

2. **内容组织原则**：
   - 使用清晰的分类头（现象、特性、型号等）
   - 分类下列出子项
   - 用自然语言描述完整信息

3. **可读性要求**：
   - 单行不超过120字符
   - 段落分明，用空行分隔
   - 使用冒号建立逻辑关系

### 转换示例

#### 示例1：故障排查表格转换

**原Markdown表格**（禁用）：
```markdown
| 现象 | 原因 | 解决方案 |
|-----|-----|--------|
| 电源灯不亮 | 电源线未连接 | 检查电源线 |

| 启动缓慢 | BIOS自检 | 等待 |
```

**转换后的完整句式**（推荐 ✅）：
```markdown
现象：电源灯不亮
电源灯不亮的可能原因及解决方案如下：电源线未连接时，需检查电源线连接；

现象：启动缓慢
启动缓慢的可能原因及解决方案如下：BIOS自检耗时，需等待或检查硬件配置。
```

## 💻 代码块规范

### 代码块格式

使用三个反引号 + 语言标识：

````markdown
```python
def hello_world():
    print("Hello, World!")
```
````

### 支持的语言标识

- `python` - Python代码
- `bash` / `shell` - Shell脚本
- `json` - JSON数据
- `yaml` - YAML配置
- `xml` - XML标记
- `html` - HTML代码
- `sql` - SQL查询
- `text` / `plain` - 纯文本（无高亮）

### 代码块规范要求

1. **必须指定语言标识**（除非是纯文本）
2. **代码缩进**：保持原始缩进结构，不额外调整
3. **长度**：过长的代码块（>50行）可考虑：
   - 分解为多个小块
   - 使用"省略号"表示中间内容：`...`
4. **特殊场景**：
   - 配置文件：使用对应格式 `yaml`, `xml` 等
   - 命令行：使用 `bash`
   - 输出结果：使用 `text` 或 `plain`

### 代码块示例

```bash
# 命令行示例
cd /path/to/directory
ls -la
```

```json
{
  "documents": [
    {
      "id": "doc_001",
      "title": "文档标题"
    }
  ]
}
```

## 🔗 链接和引用规范

### 内部链接

使用相对路径：

```markdown
[链接文本](../path/to/document.md)
[参考文档](./related.md)
```

### 外部链接

```markdown
[链接文本](https://example.com)
```

### 锚点链接

```markdown
## 目录
[跳转到第2章](#2-准备工具)

## 2 准备工具
内容...
```

### 脚注

```markdown
这是一个脚注的例子[^1]

[^1]: 脚注的具体内容
```

## 🖼️ 图片和媒体规范

### 图片格式

```markdown
![图片描述](./images/filename.png)
```

### 图片命名规范

- 文件名：使用英文和数字，用下划线或连字符分隔
- 格式：`功能_序号.png` 或 `章节_描述.png`
- 示例：
  - `installation_step_01.png`
  - `hardware_diagram.png`
  - `01产品概述_规格对比.png`

### 图片存储位置

```
TaiShan-200服务器资料/
├── 转换文档/
│   ├── 01_文档名.md
│   └── images/
│       ├── 01_图1.png
│       └── 01_图2.png
```

### 图片规范要求

1. **格式**：PNG、JPG、GIF（推荐PNG）
2. **大小**：每张图片<5MB，最佳<1MB
3. **清晰度**：>=300px宽度，避免模糊
4. **描述**：图片必须有alt文本描述

## 📄 文档长度和分割规范

### 单个文件建议

- **最小长度**：不少于200行（过短可合并）
- **最大长度**：不超过2000行（过长应分拆）
- **适宜长度**：500-1500行，便于阅读和维护

### 文件分割标准

如果单一文档超过2000行，应考虑分拆为：
- 主文档（概述、基础内容）
- 附录文档（详细参考、高级配置）
- 补充文档（故障排查、常见问题）

---

## 🔄 文本流畅性规范

### 段落规范

1. **段落之间**：使用一个空行分隔
2. **段落长度**：建议100-300字
3. **句子长度**：避免过长句子，建议<50字

### 列表和段落的混搭

```markdown
这是一个段落的开始...

1. 有序列表开始
2. 列表项

这是段落继续的部分...

- 无序列表项
- 无序列表项
```

## ✅ 审查清单

在完成每个Markdown文件转换后，使用以下清单检查：

- [ ] Frontmatter完整且格式正确
- [ ] 有且只有一个一级标题（#）
- [ ] 标题层级逻辑清晰（##, ###, ####）
- [ ] 列表格式统一（- 或 *）
- [ ] **表格已转换为语义列表完整句式**（禁止使用传统Markdown表格）
- [ ] **转换的语义列表清晰易读**（选择合适的层次：简单对应→完整句式→列表定义）
- [ ] 代码块指定语言标识
- [ ] 有特殊符号的地方已转义
- [ ] 内部链接正确（相对路径）
- [ ] 图片路径正确，有alt文本
- [ ] 没有多余空行（>2个连续空行）
- [ ] 链接可用（如有外链）
- [ ] 内容与原PDF基本保持一致
- [ ] 没有BOM或特殊字符问题

## 🎯 转换流程示意

```
原PDF文件
  ↓
提取纯文本内容
  ↓
组织章节结构
  ↓
添加Frontmatter元数据
  ↓
应用格式规范
  ├── 标题层级
  ├── 列表格式
  ├── 表格对齐
  └── 代码块标识
  ↓
生成Markdown文件
  ↓
验证并审查
  ├── 内容完整性
  ├── 格式规范性
  └── 可读性
  ↓
保存到指定位置
```

## 📦 输出文件命名规范

### 命名格式

```
{序号}_{简称}_{补充说明}.md

### 命名规则

- **序号**：01, 02, 03... (对应工作规划中的文档优先级)

```
