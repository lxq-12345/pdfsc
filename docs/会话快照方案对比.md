# 会话快照方案对比分析

**分析日期**：2026-03-11
**分析者**：Claude Code

---

## 方案A：混合方案（手动触发快照 + 自动缓冲）

### 工作流程
```
Claude回复 → 自动追加到 .snapshot_buffer.md
      ↓
用户说"快照" → Claude立即执行
      ↓
生成 mem/snapshot/快照-YYYY-M-D-HHMM.md
      ↓
清空缓冲文件，继续记录
```

### 文件结构
```
mem/
├── .snapshot_buffer.md     # 缓冲区（自动更新）
└── snapshot/
    ├── 快照-2026-3-11-1030.md
    ├── 快照-2026-3-11-1045.md
    └── 快照-2026-3-11-1100.md
```

### 优点  ✅
1. **100%可靠**
   - 不依赖后台进程
   - 每次快照都是同步生成，不会丢失
   - 已验证可正常工作

2. **实现简单**
   - 只需一个快照生成脚本（已有）
   - 用户触发快照时执行 `python scripts/gen_snapshot.py`
   - 无需系统配置、无依赖

3. **灵活控制**
   - 用户决定快照时机（关键节点快照）
   - 可以随时手动快照防止丢失
   - 可以快速快照（1分钟）或延迟快照（2小时）

4. **易于调试**
   - 每次快照过程同步进行，容易看到结果
   - 如果有问题立即反馈
   - 日志清晰可追溯

5. **跨平台**
   - Windows、WSL、Linux都支持
   - 无需系统权限配置
   - 与现有项目框架无冲突

### 缺点  ❌
1. **需要用户主动触发**
   - 用户需要记得说"快照"
   - 如果忘记快照，可能丢失内容
   - 依赖用户的执行纪律

2. **在意外中断时有风险**
   - 如果用户没有及时说"快照"就断开连接
   - 会丢失最后一段内容（最多15分钟）
   - 需要用户养成习惯（重要时刻快照）

3. **工作流中断**
   - 需要停下来说"快照"
   - 打断讨论节奏
   - 可能遗漏一些即时的快照机会

### 实现复杂度
- **初期**：低（已基本实现）
- **维护**：低（无后台管理）
- **扩展**：中（可添加自动提醒）

### 最坏情况恢复
```
会话断开（未快照）
  ↓
缓冲文件中有内容但未转换为快照
  ↓
手动执行快照生成脚本
  ↓
恢复所有宝贵的缓冲内容
```

### 成本
- 💰 零成本
- ⏱️ 每次快照 < 1秒
- 🔧 无系统配置

## 方案B：系统定时任务（Windows Task Scheduler / WSL cron）

### 工作流程

#### Windows系统
```
Windows Task Scheduler
  ↓ 每15分钟触发一次
  ↓
执行脚本：scripts/gen_snapshot.py
  ↓
自动检查缓冲，有内容则生成快照
  ↓
清空缓冲，等待下次触发
```

#### WSL系统
```
crontab -e
  ↓ */15 * * * * cd /path/to/pdfsc && python scripts/gen_snapshot.py
  ↓
每15分钟系统自动执行一次
  ↓
快照自动生成，无需用户干预
```

### 文件结构
```
mem/
├── .snapshot_buffer.md     # 缓冲区
└── snapshot/
    ├── 快照-2026-3-11-1030.md （自动生成）
    ├── 快照-2026-3-11-1045.md （自动生成）
    ├── 快照-2026-3-11-1100.md （自动生成）
    └── ...持续生成
```

### 优点  ✅
1. **完全自动**
   - 用户无需记得快照
   - 每15分钟自动生成一次
   - 不打断讨论流程

2. **风险最低**
   - 即使会话突然断开，最多丢失15分钟内容
   - "自动保险机制"
   - 频繁备份（4次/小时）

3. **无需用户干预**
   - 一次配置，永久有效
   - 对用户透明
   - 用户只需专注项目工作

4. **可追溯性好**
   - 每个时间点都有快照
   - 可以看到项目的完整时间线
   - 便于恢复到任意时刻

5. **符合协议要求**
   - 会话管理协议要求"每15分钟自动快照"
   - 这是最符合原规范意图的解决方案

### 缺点  ❌
1. **需要系统级配置**
   - Windows：需要创建Task Scheduler任务
   - WSL：需要编辑crontab
   - 初期配置复杂（20-30分钟）

2. **跨平台支持不同**
   - Windows和WSL配置方式完全不同
   - 如果切换系统需要重新配置
   - 需要了解系统调度工具

3. **依赖系统可用性**
   - Windows：后台服务必须运行
   - WSL：cron守护进程必须运行
   - 如果系统休眠/关闭，定时任务停止

4. **调试困难**
   - 如果定时任务失败，可能不知道
   - 日志在系统层面，不容易查看
   - 需要检查系统事件查看器或cron日志

5. **快照过度**
   - 每15分钟必然生成快照（即使内容重复）
   - 可能产生很多内容相同的快照文件
   - 磁盘占用（虽然不大）

6. **当前环境不确定**
   - 不清楚用户的Windows/WSL配置
   - Task Scheduler可能因权限或系统设置受限
   - cron可能在某些WSL发行版中不可用

### 实现复杂度
- **初期**：中（需系统配置）
- **维护**：低（配置好后自动运行）
- **扩展**：低（改一个定时参数）

### 最坏情况恢复
```
系统定时任务未运行（系统休眠/关闭）
  ↓
缓冲文件中有未快照的内容
  ↓
用户不知道，内容可能丢失
  ↓
（无恢复机制）
```

### 成本
- 💰 初期配置 20-30分钟
- ⏱️ 每15分钟自动执行（用户无感知）
- 🔧 需要配置系统定时任务

## 四、场景分析

### 场景1：高强度协作讨论
**特征**：频繁交互，快速迭代
- **方案A**：⚠️ 容易忘记快照，不推荐
- **方案B**：✅ 自动快照，完全不打断，推荐

### 场景2：深度开发工作
**特征**：长时间代码开发，偶尔停下讨论
- **方案A**：✅ 关键时刻快照，用户可控，推荐
- **方案B**：✅ 自动兜底保险，也不错

### 场景3：系统不稳定（网络易断）
**特征**：经常中途断线
- **方案A**：⚠️ 容易丢失未快照的内容
- **方案B**：✅ 自动频繁备份，最安全

### 场景4：脚本开发测试
**特征**：执行多个脚本，需要追踪状态
- **方案A**：✅ 每个脚本完成时快照，清晰
- **方案B**：✅ 自动快照也能接受

## 五、推荐方案

根据本项目的特点：

### ✅ 推荐：**方案B（系统定时任务）**+ **方案A兜底**

**混合策略**：
```
主机制：系统定时任务（每15分钟自动快照）
备用机制：用户可随时说"快照"手动触发

优点：
- 自动快照，无需用户记得
- 关键时刻可额外手动快照
- 双重保险，风险最低
- 符合会话管理协议要求
```

### 实现步骤
1. **立即**：保留方案A（手动触发）作为现有方案，继续工作
2. **今天**：由你配置系统定时任务（Windows Task Scheduler 或 WSL cron）
3. **后续**：自动快照运行，用户可按需手动补充快照
