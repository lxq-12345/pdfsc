概述
 2025 年，在 OpenAI ChatGPT 的指导帮助下，投入了约 1 - 2 个月时间开发了 Slice 项目，用于对某公司IT运维的批量 PDF 文件进行 RAG 知识库预处理。该工具的主要功能是：将批量 PDF文件转换为 Word 格式，再自动进行内容切片与标注，以便后续向量化与检索。
 然而，Slice 项目的预处理结果存在明显不足，例如：
 oo 文档格式失真；
 oo 部分内容缺失；
 oo 转换后的文档内容针对原文件保真度不稳定可靠。
 为解决上述问题，计划借助 Anthropic Claude 等先进 AI 工具，重新设计与实现批量 PDF 文件的预处理流程，以提高文档转换质量、保持版面结构一致性，并确保内容的完整性与可控性。
 这个新项目命名为："PDF文件预处理"，项目的根目录是D:\usr\百信公司项目\pdf_to_kbase。

 2. 确定工作范围
   打开的文件夹就是 Claude Code 的工作边界，它只会在这个范围内读取、修改、创建文件，不会误操作其他项目的文件。
 3. 建立以项目根节点为基础软件工程各阶段文档保存机制
   需求阶段：读取需求文档
   规划阶段：分析现有代码结构
   实施阶段：在正确目录下修改和创建文件
   审查阶段：对照项目文件验证结果
 4. 项目Git 版本管理的基础
   打开项目文件夹后，Claude Code 会自动识别该目录是否是一个 Git 仓库，这对整个开发过程至关重要：
   对应软件工程各阶段：
   需求阶段：通过 git log 了解项目历史，知道项目从哪里来
   规划阶段：了解当前分支状态，规划在哪个分支上开发
   实施阶段：每完成一个功能自动 commit，记录开发轨迹
   审查阶段：通过 git diff 对比变更，清晰看到修改了什么
   总结阶段：git log 形成完整的开发历史记录
   更深层的意义：
   安全保障：每次修改前有 commit 记录，出错可以随时回滚
   协作基础：多人开发时通过 Git 协调工作，避免冲突
   与 GitHub 连接：本地项目通过 Git 推送到 GitHub，实现代码备份和共享
   Claude Code 的 checkpoint 机制：也是基于 Git 原理，每次重要操作前保存状态，随时可以 rewind
 5. CLAUDE.md 的加载
   打开项目文件夹后，Claude Code 会自动读取目录下的 CLAUDE.md 文件，获取项目的背景信息、开发规范和历史约定，这相当于给 Claude 的项目说明书。

Markdown转换规范.md文件内容简介
 "Markdown转换规范.md"文件是Claude Agent在了解了项目资料内容，在项目规划任务中创建的一份PDF格式转换Markdown的格式的规范文档：
 主要内容包括：
 1. Frontmatter元数据规范
   所有Markdown文件必须以YAML格式的Frontmatter开头
   包含title、category、type、model、version、date、keywords、source_pdf等字段
 2. 标题层级规范
   规定一级标题（#）到四级标题（####）的使用方式
   禁止使用五级及以上标题
   每个文件只能有一个一级标题
 3. 列表规范
   无序列表、有序列表、定义列表的格式要求
   明确缩进规则（2空格、4空格等）
   表格规范 ⭐ 核心内容
   禁用传统Markdown表格，改用语义列表完整句式
   提供三层次递进方案：
   第一层：简单语义列表（3行以内）
   第二层：完整句式格式（推荐，最佳RAG理解效果）
   第三层：列表加定义格式（复杂多维数据）
 4. 代码块规范
   必须指定语言标识（python、bash、json等）
   保持代码原始缩进
 5. 其他规范
   链接和引用规范
   图片和媒体规范
   重点标记（粗体、斜体、删除线）
   特殊格式（提示框、分隔线）
   文档长度建议（500-1500行）
   文本流畅性规范
   文件命名规范
 6. 审查清单
   转换完成后逐项检查的清单
Markdown转换规范.md文件与 Skill
 "Markdown转换规范.md" 又是一份非常完整的 Skill 文档，它体现了 Skill 的本质，完全符合 Skill 的定义 --  -- 它不是泛泛的规范说明，而是给 AI 的可执行操作手册，体现在以下几个方面：
 1. 明确的触发场景：PDF --> Markdown 转换任务，适用范围清晰（TaiShan-200知识库项目）。
 2. 完整的操作流程：文档末尾有完整的转换流程图，从"提取纯文本"到"验证审查"，AI 按步骤执行即可。
 3. 具体的格式规则，覆盖所有转换场景：
   Frontmatter 元数据模板（必填字段、格式示例）
   标题层级规范（最多四级，禁止五级以上）
   表格处理规则（这是最核心的规则 --  -- 禁用传统 Markdown 表格，改用语义列表完整句式，原因是为了优化 RAG 向量化检索效果）
   代码块、链接、图片、文本标记等格式规范
 4. 审查清单：转换完成后有 14 项 checklist，AI 可自检输出质量。
 5. 具体的转换示例：提供"原始表格 --> 转换后格式"的对比示例，AI 可以直接参照执行。
 最值得关注的设计决策
 文档中最有价值的规则是废弃传统 Markdown 表格，改用三层次的语义列表格式。这个决策背后有明确的技术理由：传统表格在 RAG 向量化后语义理解差，而完整句式的自然语言描述能让模型更准确地检索和理解内容。
 这说明这份 Skill 不只是格式规范，而是针对下游 AI 使用场景（知识库检索）专门优化过的操作规则 --  -- 这正是一份高质量 Skill 文档的特征。
任务实施
 完成项目规划后进入到项目任务实施阶段，需要将工作模式调整为：编辑...。
编辑
 项目任务实施的核心动作是不断修改文件。写代码是在编辑 .py 文件，改配置是在编辑 config.yaml，写文档是在编辑 .md 文件。所以"编辑"这个词抓住了项目开发的本质 --  -- 持续地、有目的地修改一组项目文件。
 Claude Code 之所以适合项目任务，正是因为它有真正的"编辑"能力：能创建或读取项目本地文件，知道文件现在长什么样，然后精确地修改其中某一部分，这就像一个真正在用户电脑前工作的助手，而不是一个只会口头建议的顾问。
 相比之下，网页 Claude.ai 没有这种能力。它每次都是"说给你听"，用户还需要自己去复制粘贴、手动改文件。对于偶尔的问答没问题，但如果要处理一个有几十个文件的项目，来回复制粘贴就非常低效了。
 所以简单说：问答靠"说"，项目靠"做"，"编辑"就是"做"的核心动作。
 
 1. 编辑前询问
   Claude 每次要修改/创建项目或程序文件时
   oo 先弹出确认框
   oo 要求确认"Allow"才执行任务步骤，或者"Skip"跳过任务步骤。
   oo 相当于每步涉及文件创建或内容修改的操作"都要审批"。
 2. 自动编辑
   Claude 直接修改文件
   oo 不询问不确认
   oo 全自动执行
   oo 相当于"全权委托"
 注：我在实操中，发现选择"自动编辑"未能实现"一键式"全自动任务实施，还是不断有确认要求中断任务执行，需要我选中"Allow"。
 对于这个情况，Claude给予解释：
 根本原因是人类监督（Human-in-the-Loop）机制。Claude Code 即使在"自动编辑"模式下，也不会对某些高风险操作完全放权。每次弹出 Allow 确认，本质上是系统在说："我接下来要做一件可能影响你文件/系统的事，你确认吗？"
 这背后有几个现实考虑：
 第一，AI 会犯错。如果全自动执行，一个判断失误可能导致文件被错误覆盖、代码被破坏性修改，而你完全不知情。确认步骤是一道安全网。
 第二，操作是不可逆的。读文件可以随意，但写文件、删文件、执行命令这类操作一旦完成很难撤销，所以系统对这类操作格外谨慎。
 第三，"自动编辑"的含义被误解了。它的"自动"指的是 Claude 可以自主决定编辑哪里、怎么编辑，而不是"绕过所有确认"。确认机制是独立于编辑决策之外的安全层。
 所以本质上，当前 AI Agent 的设计哲学是：让 AI 自主思考和规划，但保留人类对关键执行步骤的最终控制权。"全自动"在当前阶段更多是一个理想状态，而不是现实产品的实际行为。
