## 一、概述

### 1.1 问题背景

在AI项目开发中，会话经常因以下原因而中断和丢失内容：
- 网络连接不稳定断线
- Claude进程意外崩溃
- 长时间对话导致上下文丢失
- 用户误操作导致会话关闭

这些中断往往导致：
- 已完成的工作成果无法追溯
- 需要重新描述背景和历史
- 效率下降，时间浪费

为了解决这个问题，需要一套自动化的会话保存和恢复机制。

### 1.2 解决方案

设计了一套基于"配置驱动"思想的完整会话快照系统：

配置定义规则
会话管理协议.md 定义快照触发规则和工作流程

脚本自动执行
snapshot_trigger.sh 或 snapshot_handler.py 自动读取规则并执行

用户友好触发
用户只需说"快照"即可自动保存会话内容

逐步积累保存
快照文件按时间戳逐步积累，便于后续备忘整理

### 1.3 核心特点

✅ **配置驱动**
规则在文档中定义，修改规则无需改代码，直接改协议文件即生效

✅ **用户友好**
只需说"快照"即可保存，完全自动，无需学习复杂命令

✅ **可追溯性**
每个快照带时间戳，能精确追踪会话历史，了解工作进展

✅ **可靠性**
防止会话意外中断丢失内容，最多只丢失最后一个快照之后的部分

✅ **零维护成本**
一次配置，多次使用，无需后续调整

## 二、快速开始（5分钟入门）

### 2.1 系统要求

操作系统
Windows 11 + WSL2 / Linux / macOS

Shell环境
Bash（推荐用 snapshot_trigger.sh） 或 Python3（用 snapshot_handler.py）

文件存储
至少 100MB 空间用于快照文件积累

网络
Claude 在线，网络连接稳定（对快照本身无要求）

### 2.2 最常用的三个场景

#### 场景1：启动新会话

新打开 Claude 会话时

第一步：等待系统提示

第二步：说"快照"

#### 场景2：保存工作进度

会话进行中，任何时刻都可以说

常见时机：
- 完成一个重要任务后
- 准备做不同的工作前
- 定时保存（建议每15-30分钟一次）
- 网络不稳定时

操作步骤：

第一步：说"快照"

用户说："快照"

第二步：等待确认

一天工作全部完成，准备结束会话时

- 文件变更

### 2.3 三个关键词（必记）

【快照】
命令：说"快照" 或 "保存快照"
作用：立即保存本次会话的所有内容到快照文件
时机：任何时刻都可以说，建议15分钟说一次
输出：✅ 快照已生成: mem/snapshot/快照-YYYY-M-D-HHMM.md
恢复：中断后可从该快照点继续，最多丢失快照后的内容

【归档】
命令：说"归档" 或 "结束会话"
作用：将当天所有快照整理为备忘文件
时机：一天工作结束时说一次
输出：✅ 已生成备忘-YYYY-M-D.md
产出：结构化的会议纪要式备忘文件

【继续】
命令：说"继续pdfsc" 或 "恢复上一个session"
作用：恢复上一个会话的上下文
时机：会话意外中断后，重新启动时说
输出：✅ 已自动加载最新快照，恢复上下文完毕
效果：无需重新描述背景，直接继续工作

## 三、使用指南（详细说明）

### 3.1 会话启动流程

完整的启动步骤说明

第一步：启动 Claude 会话

打开 Claude Code 并开始新会话。

系统自动执行以下操作：
- 检查 mem/snapshot/ 目录中的最新快照文件
- 读取快照的日期
- 如果上一个会话的快照日期早于今天，自动生成当天的备忘文件
- 初始化缓冲文件 mem/.snapshot_buffer.md
- 准备接收新的会话内容

第二步：提示用户说"快照"

系统提示：
请说"快照"来保存上一个会话的内容

用户说："快照"

第三步：生成上一个会话的快照

系统响应：✅ 快照已生成: mem/snapshot/快照-2026-03-11-1130.md

此时：
- 上一个会话的所有内容已保存到快照文件
- 缓冲文件已清空
- 新的缓冲文件已初始化
- 准备记录本次会话的内容

第四步：开始正常工作

从这一刻开始，所有对话都被记录到缓冲文件中。
无需做任何额外操作，系统会自动保存。

标记重要节点

定时保存
推荐每隔15-30分钟说一次"快照"，确保内容安全

操作方式：

等待系统响应："✅ 快照已生成: mem/snapshot/快照-YYYY-M-D-HHMM.md"

### 3.4 会话中断恢复

如果 Claude 会话意外中断，下次启动时如何恢复

会话中断的情况：
- 网络突然断开
- Claude 进程崩溃
- 浏览器标签页关闭
- 电脑休眠/重启
- 其他意外原因

恢复流程：

第一步：启动新会话

重新打开 Claude Code

等待启动完毕（系统自动执行启动检查）

第二步：说"继续pdfsc"

用户说："继续pdfsc"

或说："恢复上一个session"

第三步：自动恢复上下文

系统执行：
① 查找最新的快照文件
② 加载快照内容到上下文
③ 恢复之前的工作状态

系统响应：
✅ 已自动加载最新快照
✅ 恢复上下文完毕
请继续之前的工作...

第四步：继续工作

直接继续之前的任务，无需重新描述背景。

注意事项：

恢复的范围
从最新快照点之后的内容需要自己回忆或补充

丢失的内容
最后一个快照之后、中断前的对话内容无法恢复
（这就是为什么要定时说"快照"）

如何最小化丢失
1. 定时说"快照"（建议15分钟一次）
2. 重要任务完成后立即说"快照"
3. 网络不稳定时频繁说"快照"

## 四、快照机制详解（技术原理）

### 4.1 整体执行流程

从启动到保存的完整流程图

会话启动
  ↓
【步骤0】启动检查与初始化
  操作：
  - 读取 mem/snapshot/ 最新快照文件日期
  - 如果上次未归档，生成备忘文件
  - 初始化 mem/.snapshot_buffer.md（缓冲文件）
  - 记录 session_start_time = NOW()
  ↓
【步骤1】配置加载
  操作：
  - 读取 会话管理协议.md
  - 解析第四章"触发词汇表"
  - 提取关键词：["快照", "保存快照", "归档", "结束会话", "继续", ...]
  - 缓存到内存供后续使用
  ↓
【步骤2】会话进行中 - 正常对话循环
  操作：
  - 用户输入消息
  - Claude 生成回应
  - 两者追加到缓冲文件
  - 等待用户下一条消息
  循环进行，每条消息都被记录到 .snapshot_buffer.md
  ↓ （用户说"快照"或"归档"时触发）
【步骤3】关键词识别
  操作：
  - 检查用户消息是否包含快照关键词
  - 使用正则匹配：grep -E "快照|保存快照|归档|结束会话"
  - 如果"快照" ✅ → 执行快照操作（步骤4）
  - 如果"归档" ✅ → 执行备忘操作（步骤4B）
  - 都不匹配 → 继续正常对话（回到步骤2）
  ↓
【步骤4】快照生成
  操作：
  a) 读取缓冲文件内容 (mem/.snapshot_buffer.md)
  b) 获取当前时间戳：now = datetime.now()
  c) 构造快照文件名：快照-YYYY-M-D-HHMM.md
  d) 将缓冲内容写入：mem/snapshot/快照-YYYY-M-D-HHMM.md
  e) 清空缓冲文件
  f) 重新初始化缓冲（开始新周期）
  g) 向用户通知："✅ 快照已生成"
  ↓
【步骤5】继续会话或结束
  分支：
  - 如果用户继续对话 → 回到步骤2
  - 如果用户说"归档"  → 进入会话结束流程

## 五、配置驱动原理（设计思想）

### 5.1 配置驱动 vs 硬编码

什么是硬编码方式（不好的做法）

在代码中写死规则：

if user_message == "快照":  # 硬编码
    generate_snapshot()
elif user_message == "保存快照":  # 硬编码
    generate_snapshot()
elif user_message == "归档":  # 硬编码
    generate_memo()

问题：
❌ 改规则必须改代码
❌ 改代码必须重新部署
❌ 维护困难，容易出错
❌ 不灵活，扩展性差

什么是配置驱动方式（好的做法）

规则定义在文档中：

文件：会话管理协议.md

快照触发：    "快照" / "保存快照"
结束归档：    "归档" / "结束会话"

脚本读取并执行：

SNAPSHOT_KEYWORDS=$(grep "快照触发：" "会话管理协议.md" | ...)
if echo "$user_message" | grep -qE "$SNAPSHOT_KEYWORDS" then
    generate_snapshot()
fi

优势：
✅ 改规则只需改文档，无需改代码
✅ 改文档立即生效，下次会话自动加载
✅ 灵活可扩展，易于维护
✅ 配置和代码分离，职责清晰

### 5.2 修改规则的方式（示例）

如果需要修改快照关键词

示例1：添加新的快照关键词

原规则（会话管理协议.md）：
快照触发：    "快照" / "保存快照"

需求：增加"save"作为快照关键词

修改后：
快照触发：    "快照" / "保存快照" / "save"

下次会话启动时，脚本自动读取新规则。

示例2：修改触发关键词

原规则：
快照触发：    "快照" / "保存快照"

需求：改为"s"（更简短）

修改后：
快照触发：    "s" / "保存进度"

下次会话启动时，用户可以说"s"代替"快照"。

示例3：增加新的触发动作

     保存快照

步骤③ tr '\n' '|'
作用：将换行符替换为竖线（用作分隔符）
输入：快照
     保存快照
输出：快照|保存快照

步骤④ sed 's/|$//'
作用：删除末尾的竖线
输入：快照|保存快照|

## 八、故障排查（问题解决）

### 8.1 快照文件未生成

现象：用户说"快照"但没有生成快照文件

排查步骤：

第一步：检查关键词配置是否正确

执行命令：
grep "快照触发：" 会话管理协议.md

预期输出应包含：
快照触发：    "快照" / "保存快照"

如果输出为空或格式错误：
→ 检查协议文件内容
→ 确保格式完全一致
→ 重新启动会话让脚本重新加载

第二步：检查缓冲文件是否有内容

执行命令：
ls -lh mem/.snapshot_buffer.md

输出示例：
-rw-r--r-- 1 user group 2.5K mem/.snapshot_buffer.md

检查项：
- 文件是否存在（应该存在）
- 文件大小（应该 >100 字节，说明有对话内容）

如果文件为0字节：
→ 说明缓冲为空，没有对话内容记录
→ 继续对话，产生内容后再试

第三步：检查快照目录权限

执行命令：
ls -ld mem/snapshot/

输出示例：
drwxr-xr-x 1 user group mem/snapshot/

检查项：
- 目录是否存在（应该存在）
- 权限最后三位是否都有 r 或 w（应该可读可写）

如果权限不足：
→ 执行：chmod 755 mem/snapshot/
→ 确保目录可写

第四步：手动测试关键词识别

执行命令：
bash scripts/snapshot_trigger.sh

预期输出：
=== 快照配置驱动处理器 ===

【1】从协议读取关键词配置
✅ 已加载规则:
   快照触发: 快照|保存快照
   结束归档: 归档|结束会话

【2】检查消息: '快照'
   ✅ 匹配: 快照触发

【3】执行快照流程测试
【4】执行快照生成
   生成快照文件: ...
   ✅ 快照已保存

如果输出异常：
→ 检查Bash环境
→ 确保 snapshot_trigger.sh 可执行
→ 运行：chmod +x scripts/snapshot_trigger.sh

### 8.2 快照文件格式错误

现象：快照文件存在但格式不符合规范

检查项：

第一步：验证快照内容格式

执行命令：
head -20 mem/snapshot/快照-2026-03-11-1133.md

正确格式应该是：
时间段：HH:MM - HH:MM

用户：消息内容
Claude：回应内容

用户：消息内容
Claude：回应内容

错误示例（不应包含）：
本次快照总结
讨论主题
关键决策
...

如果包含这些额外内容：
→ 说明混乱了快照和备忘的格式
→ 快照应该是原始对话，无加工
→ 备忘才是结构化的摘要

第二步：验证时间戳格式

执行命令：
ls mem/snapshot/ | head -5

正确格式：
快照-2026-03-11-1130.md
快照-2026-03-11-1135.md
快照-2026-03-11-1140.md

格式说明：快照-YYYY-M-D-HHMM.md
- YYYY：4位年份
- M：月份（1-12，不补零）
- D：日期（1-31，不补零）
- HH：小时（00-23）
- MM：分钟（00-59）

如果时间戳格式错误：
→ 检查系统时间：date
→ 检查脚本中的时间格式化代码
→ 可能是时区问题

### 8.3 缓冲文件持续增长

现象：缓冲文件（.snapshot_buffer.md）一直增长，没有被清空

检查项：

第一步：检查快照是否真正生成

执行命令：
ls -lt mem/snapshot/ | head -3

看最新生成的快照文件时间是否近期

如果最新快照的时间很久：
→ 说明快照生成没有被触发
→ 检查关键词识别是否正常
→ 查看是否说了"快照"

第二步：查看缓冲文件内容

执行命令：
wc -l mem/.snapshot_buffer.md

如果行数超过1000行以上：
→ 说明太久没有快照了
→ 立即说"快照"保存

第三步：手动清空缓冲

如果需要重置，执行：
rm mem/.snapshot_buffer.md

然后重新初始化：
echo "# 会话快照缓冲 - $(date +%Y-%m-%d-%H:%M)+" > mem/.snapshot_buffer.md

这样会失去未保存的内容，谨慎操作！

### 8.4 协议文件更新后规则未生效

不需要重启系统或重新部署程序。
